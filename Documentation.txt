1. PROJECT OVERVIEW

TaskFlow is a backend-focused task management application built using Django and Django REST Framework.

The system allows users to create, update, retrieve, and delete tasks through secure RESTful APIs. Each task is associated with a specific authenticated user to ensure data isolation.

In addition to the API, the project includes a lightweight Django template interface that demonstrates how users can interact with the backend through a dashboard-based workflow.

The application demonstrates key backend development concepts including authentication, API design, caching, rate limiting, secure data access, validation, and containerized local development.

4. DATABASE MODEL

The Task model contains the following fields:

title
status
priority
deadline
user
notes

Explanation:

title
Represents the name or description of the task.

status
Indicates the completion state of the task (pending or completed).

priority
Defines the importance level of the task (low, medium, high).

deadline
Stores the due date for the task.

user
ForeignKey linked to Django's built-in User model to ensure each task belongs to a specific user.

notes
Stores optional additional notes or details related to a specific task.

5. AUTHENTICATION SYSTEM

The project supports both JWT authentication for API access and session-based authentication for the Django template interface.

Session authentication enables users to log in through the web interface and manage tasks using the dashboard.

Authentication protections implemented in the project include:

вЂў Dashboard access restricted using Django's login_required decorator
вЂў Logged-in users are redirected away from login and registration pages
вЂў Logout redirects users to the login page
вЂў Browser caching is disabled on protected pages using never_cache to prevent access through the browser back button
вЂў Username uniqueness validation during registration
вЂў Confirm password validation during registration

API authentication endpoints:

POST /api/register/
Creates a new user account.

POST /api/login/
Generates JWT access and refresh tokens.

POST /api/token/refresh/
Refreshes expired access tokens.

6. TASK MANAGEMENT API

The main API resource is the Task endpoint.

Base Endpoint:

/api/tasks/

Supported operations:

GET /api/tasks/
Retrieve tasks belonging to the authenticated user.

POST /api/tasks/
Create a new task.

PUT /api/tasks/{id}/
Update an existing task.

DELETE /api/tasks/{id}/
Delete a task.

Each operation is restricted to tasks owned by the authenticated user.

---

7. FILTERING

Tasks can be filtered using query parameters.

Examples:

/api/tasks/?status=pending
/api/tasks/?priority=high

Filtering allows users to quickly retrieve specific subsets of tasks.

---

8. PAGINATION

Pagination is implemented through Django REST Framework settings to ensure efficient responses when dealing with large datasets.

This prevents large API responses and improves performance.

---

9. REDIS CACHING

Redis is used to cache frequently requested API responses.

Example:

GET /api/tasks/

When a user requests their task list, the response is cached in Redis for a short duration.

Subsequent requests retrieve the data from Redis instead of querying the PostgreSQL database again.

Cache keys are stored per user to maintain proper data isolation.

---

10. REDIS RATE LIMITING

Redis is also used to implement API rate limiting.

The system tracks the number of requests made by each user within a defined time window.

Example rule:

20 requests per minute per user

If the limit is exceeded, the API returns:

HTTP 429 вЂ“ Too Many Requests

This protects the API from abuse and excessive traffic.

---

11. USER DATA ISOLATION

Each user can only access and modify their own tasks.

This is enforced in API queries using:

Task.objects.filter(user=self.request.user)

This ensures that users cannot view or modify tasks belonging to other users.

13. INPUT VALIDATION

To maintain data quality and prevent invalid data entry, several validation rules are implemented:

вЂў Task title cannot be empty
вЂў Task title must meet minimum length requirements
вЂў Task title has a maximum length restriction
вЂў Task deadlines cannot be set in the past
вЂў Login credentials must not be empty
вЂў Registration usernames must be unique
вЂў Registration passwords must match using confirm password validation

These validations help prevent invalid or inconsistent data from being stored in the database.

14. SECURITY PRACTICES

The application implements several security practices:

вЂў Login protection for dashboard views using login_required
вЂў Cache prevention using never_cache to prevent dashboard access after logout
вЂў Safe object retrieval using get_object_or_404 with user ownership checks
вЂў User-specific filtering to prevent unauthorized data access
вЂў Validation for authentication forms to prevent empty credentials
вЂў API rate limiting to prevent abuse
вЂў Admin insights page access restricted to superusers
вЂў User deletion from admin insights restricted to superusers only

These practices ensure proper access control and application stability.
